Контекст виконання

Кожен запит, який надходить до системи, обробляється власним контекстом виконання. Цей контекст включає в себе кілька важливих елементів:

  1. Унікальний ідентифікатор запиту: Кожен запит отримує унікальний ідентифікатор, який використовується для єдиної ідентифікації та відстеження його обробки.
  2. Знімок бізнес-схеми виконання: Це представлення стану бізнес-логіки або сценарію, який виконується під час обробки запиту. Знімок містить бізнес-схему на початок виконання запиту.
  3. Інформація про сам запит: Це дані, які надходять у запиті, такі як URL, заголовки, метод запиту, параметри шляху, тіло запиту та параметри запиту.
  4. Сесію користувача чи системи: В залежності від типу приватизації контекст може містити сесію користувача або сесію системи.

Типи приватизації​

x-fiber підтримує наступні типи приватизації:

export type AuthScope = "public:route" | "private:user" | "private:system";

де:

  • public:route: Цей тип приватизації використовується для публічних маршрутів, які доступні для всіх користувачів без обмежень.
  • private:user: Цей тип приватизації застосовується до запитів, що відносяться до конкретного користувача. Для цього типу потрібні відповідні токени доступу користувача для аутентифікації та авторизації.
  • private:system: Цей тип приватизації використовується для системних запитів, які взаємодіють зі службами або компонентами системи. Для цього типу також потрібні відповідні системні токени доступу.
інформація

Залежно від типу приватизації, http адаптер ядра обчислень формує відповідний контекст виконання, що враховує наявність відповідних токенів доступу, окрім випадку з public:route.

Формування контексту​

Контекст формується відповідно до типу приватизації, але також має базову частину, яка складається зі сховища загальної інформації. Після перевірки та визначення конкретного обробника останнього етапу, адаптер ядра обчислень визначає контекст виконання. Кожен контекст виконання включає в себе store, який містить загальну інформацію про запит, IP-адресу, з якої здійснено запит, знімок бізнес-схеми та інші дані.

Базовий контекст​

Базовий контекст складається з бізнес-схеми та деталей запиту.

type BusinessScheme = {
    // ... business scheme description
}

type Store = {
  service: string, // service name
  domain: string, // domain name
  action: string, // action name
  method: string, // http method
  path: string, // full request url
  ip: string, // request ip
  requestId: string, // unique request id
  schema: BusinessScheme, // snapshot business scheme
  language: string, // supported language
}

type Context<U extends Record<string,any> = never, S extends Record<string,any> = never> = {
    store: Store
    // ... another fields
}

public:route​

Тип приватизації не передбачає перевірок щодо приватності запитів та відсутності перевірки токенів доступу, оскільки вважається, що запити з типом public:route призначені для публічних маршрутів, таких як головні сторінки, опис товарів і т. д.

Нижче наведено структуру контексту:

type Store = {
    // base store context
}

type Context = {
    store: Store
    user: undefined,
    system: undefined
}

private:user​

Тип приватизації передбачає, що запити є конфіденційними і вимагають наявності JWT токену доступу користувача під назвою x-user-access-token. Цей токен надається у двох випадках: під час процедури авторизації при вході в веб-застосунок та при оновленні завдяки x-user-refresh-token.

порада

Для створення авторизації використовуйте вбудовані методи агента fnAgent простору імен sessions, який містить методи для організації роботи з сесіями, включаючи авторизацію. Також використовуйте методи агента fnAgent простору імен scrambler для шифрування, хешування та підтвердження.

Докладний перелік методів агента функціональності дивись Агент функціональності

Під час обробки запиту з типом приватності private:user, адаптер http ядра обчислень намагатиметься знайти заголовок HTTP з назвою x-user-access-token.

1. Випадок відсутності заголовку

В цьому випадку http адаптер поверне відповідь із http статус кодом 400, що вказує на помилку. Нижче наведено приклад формату цієї відповіді:

{
  "type": "FAIL",
  data: {
    "code": "1002.1001",
    "message": "Authorization failed: miss required `x-user-access-token` http header"
  }
}

2. Випадок наявності токена зі сплинутим терміном

В цьому випадку http адаптер поверне наступну відповідь з http статус кодом 401:

{
  "type": "FAIL",
  data: {
    "code": "1002.1002",
    "message": "Authorization failed: user access token has expired."
  }
}

3. Випадок дійсного токену

У випадку, коли заголовок присутній і токен є дійсним, адаптер перевіряє jwt токен на наявність sessionId. Цей sessionId створюється під час авторизації користувача, використовуючи методи з простору імен sessions в веб-застосунку і використовується для отримання інформації про сесію користувача з Redis сховища. Після успішного отримання інформації адаптер додає ці дані до контексту виконання.

type Store = {
    // ... base request context
}

type Context<
    U extends Record<string,any> = never,
    S extends Record<string,any> = never
  > = {
    store: Store
    user: {
     userId: string // user unique identifier
     sessionId: string // session unique identifier
    } & U // other information about the user session
    system: S // never
}
примітка

Цей механізм авторизації використовується для доступу до різноманітних ресурсів, пов'язаних з профілем користувача, таких як особисті дані, інформація про ролі, налаштування та інша приватна інформація, необхідна для забезпечення безпеки та відповідності до політик конфіденційності.

private:system​

Тип приватизації передбачає, що запити є приватними і вимагають наявності jwt токену доступу до системи - x-system-access-token. Цей токен надається в трьох випадках:

  • При безпосередньому виданні токена, який може бути включений у договір або інші правові документи.
  • Під час авторизації в веб-застосунку через спеціальний маршрут, що описаний в бізнес-схемі.
  • Під час оновлення, за допомогою x-system-refresh-token, який використовується для поновлення токенів.
порада

При авторизації системи через кінцевий маршрут, використовуйте вбудовані методи агента fnAgent простору імен sessions, який містить методи для організації роботи з сесіями, включаючи авторизацію. Також використовуйте методи агента fnAgent простору імен scrambler для шифрування, хешування та підтвердження.

примітка

Зазначений тип приватизації є ключовим у забезпеченні безпеки та конфіденційності при доступі до ресурсів на маршрутах інтеграцій та побудові міжсистемної взаємодії.

Опис приватизації​

Наша концепція полягає в розробці високоякісних структур обробників запитів, які дозволяють здійснювати повний контроль над кодовою базою бізнес-схеми за допомогою TypeScript. Цей підхід дозволяє докладно описувати кожен аспект обробки запитів, забезпечуючи чіткість, надійність і легкість супроводу коду. Завдяки цьому, розробники можуть ефективно управляти та модифікувати структури обробників, щоб вони відповідали вимогам та змінам у бізнес-логіці.

  • Router
  • Types declaration
import { setRouter } from "@x-fiber/calculator";

import type { NBusUsersAgg } from "~BusinessAdmin/Domains/Types";

export const BusUsersAggRouter = setRouter<NBusUsersAgg.Routes>({
    "v1/getOne": {
        GET: {
        scope: "private:user",
        params: [ "userId" ],
        headers: ["x-access-user-token", 'application/json'],
        queries: [],
        handler: async (request, agents, context) => {
                context.user.firstName // Поле сесії;
                context.user.lastName // Поле сесії;
            },
        },
    },
});

export type BaseSessionInfo = {
    sessionId: string;
    userId: string;
};

export type Routes = {
    "v1/getOne": {
        GET: NAbstractHttpAdapter.Handler<
            void // тіло запиту,
            void // параметри запиту,
            void // заголовки запиту,
            void // query параметри запиту,
            void // тіло відповіді запиту,
            void // заголовки відповіді запиту,
            BaseSessionInfo & { firstName: string; lastName: string } // тип сесії користувача,
            void // тип сесії системи,
            "private:user" // тип приватизації  "public:route" | "private:user" | "private:organization";
            >;
    };
};
