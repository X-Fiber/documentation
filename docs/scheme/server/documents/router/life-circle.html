Життєвий цикл

Обробка http запитів здійснюється http адаптером HttpAdapter ядра обчислень, який формує життєвий цикл запиту. Кожен обробник запиту пройде через ряд етапів у своєму життєвому циклі. Ці етапи включають перевірку CORS, валідацію схеми, створення контексту та виконання обробника та ін. Повний життєвий цикл виглядає наступним чином:

1. Перевірка CORS Механізм CORS доступний для використання. Додатковий опис може бути наданий при описі конкретного сервісу.

2. Перевірка заголовків Валідація елементів маршруту, включаючи побудову об'єкта динамічних параметрів та перевірку наявності обов'язкових заголовків. Також проводиться перевірка структури параметрів запиту. У випадку відсутності обов'язкових елементів, http адаптер HttpAdapter ядра обчислень сформує наступну відповідь зі статусом 400:

{
    "type": "FAIL",
    "data": {
        "code": "1001.0001",
        "message": "Miss required header {header-name}. " // for header not found exception
    }
}

3. Перевірка елементів маршруту Перевірка валідності елементів маршруту. Пошук відповідного сервісу, прикладної області та маршруту з HTTP запитом. У випадку, якщо відповідний елемент маршруту не знайдений, веб-сервер сформує наступну відповідь зі статусом 400:

{
    "type": "FAIL",
    "data": {
        "code": "1000.0001",
        "message": "Service {service-name} not found." // for service not found exception
    }
}
небезпека

Ядро обчислень підтримує опцію development mode, яка в разі відсутності маршруту повертає перелік можливих REST API методів з урахуванням прикладних областей та сервісів.

4. Перевірка підтримуваної мови

Виконується перевірка підтримуваної мови для обробки запиту з відповідною мовою. При налаштуванні конфігурації веб-застосунку потрібно вказати список підтримуваних мов, які будуть використовуватися для перевірки.

{
  "services": {
    "localization": {
      "defaultLanguage": "en",
      "fallbackLanguage": "ru",
      "supportedLanguages": ["en", "ru", "ua"]
    }
  }
}

Для кожної підтримуваної мови необхідно створити відповідний словник, який буде використовуватися при обробці запиту з конкретною мовою перекладу.

import { setDictionary } from "@x-fiber/calculator";

type LanguagesKind = 'uk' | 'uk-UK' | 'en' | 'en-EN' | 'sp' | 'sp-SP'

export type Dictionary = {
    create: {
        success: string
    }
    title: {
        init: string
    }
    actions: {
        getOne: string
    }
}

const languages = ['en', 'en-EN']

export const BusUsersAggDictEn = setDictionary<
  LanguagesKind,
  Dictionary
>(languages, {
  create: {
    success: "User successful create",
  },
  titles: {
    init: 'Initiator'
  },
  actions: {
    getOne: 'Get user by unique identifier'
  }
});
інформація

Деталі створення словників визначається документом "Словник". Деталі роботи документа дивись Словник

5. Перевірка простору авторизації AuthScope

Простору авторизації AuthScope визначає доступність ресурсу для користувачів та систем на основі їхніх авторизаційних привілеїв. Існують 3 типи просторів авторизації, які визначають рівень доступу:

export type AuthScope = "public:route" | "private:user" | "private:system"
ТипПриватністьТокен доступуОпис
public:route--Публічні маршрути, які не потребують перевірки на будь-які токени доступу. Використовуються для бізнес-логіки, яка не потребує реєстрації та авторизації користувачів.
private:user+x-user-access-tokenПриватні маршрути, які потребують авторизації користувача. Використовуються для бізнес-логіки перевірки за ролю, а також для роботи з сесією користувача та ін.
private:system+x-system-access-tokenПриватні маршрути, які потребують авторизації системи. Використовуються для бізнес-логіки перевірки системи при побудові інтеграцій та ін.
інформація

Деталі обробки простору авторизації, від якого напряму залежить обʼєм контексту виконання обробника запиту дивись Контекст виконання

6. Формування контексту

Контекст запиту складається зі знімку версії, бізнес-схеми, інформації про маршрут, а також обʼєкти сесій.

type BusinessScheme = {
    // ... business scheme description
}

type Store = {
  service: string, // service name
  domain: string, // domain name
  action: string, // action name
  method: string, // http method
  path: string, // full request url
  ip: string, // request ip
  requestId: string, // unique request id
  schema: BusinessScheme, // snapshot business scheme
  language: string, // supported language
}

type UserContext<U> = {
    // ... user context description
}

type SystemContext<S> = {
    // ... system context description
}

type Context<U extends Record<string,any> = never, S extends Record<string,any> = never> = {
    store: Store
    user: UserContext<U>
    system: SystemContext<S>
}
warning

В залежності від типу приватності контекст буде по різному доповнений. Деталі формування структур залежних від типу приватності дивись Контекст виконання

7. Виконання контексту запиту

Створює новий контекст для зберігання даних у AsyncLocalStorage для забезпечення ізоляції даних між різними запитами або операціями, що теоретично можуть відбуватись паралельно, з привʼязуванням до унікального ідентифікатор запиту.

8. Виконання запиту бізнес-схеми

Здійснюється виклик обробника кінцевого маршруту з передачею агентів доступу, контексту запиту та даних запиту з можливих поверненням результату.

9. Формування відповіді

В залежності від типу результату відбувається формування відповіді на запит.
