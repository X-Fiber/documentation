Контекст виконання

Кожний запит виконується, в рамках власного контексту з включенням унікального ідентифікатора запиту, знімку бізнес-схеми виконання та інформації про сам запит.

Типи приватизації​

x-fiber підтримує наступні типи приватизації:

export type AuthScope = "public:route" | "private:user" | "private:system";
інформація

В залежності від типу приватизації сервер обчислень формує різний контекст виконання. Це звʼязано з тим, що кожний контекст передбачає наявність відповідних токенів доступу (окрім public:route).

Формування контексту​

Після валідації та визначення конкретного обробника кінцевого маршруту адаптер ядра обчислень вираховує контекст виконання. Кожний контекст виконання враховує store, який містить загальну інформацію про запит, ip адресу запиту, знімок бізнес-схеми та інше.

const context = { // структура обʼєкту контексту виконання
    store: store, // структура store
    user: {}, // інформація про сесію користувача, наявна коли тип приватизації 'private:user' чи 'private:organization'
    system: {}, // інформація про сесію системи, наявна коли тип приватизації 'private:system'
}

const store = {  // загальна структура store
    service: service, // назва сервісу
    domain: domain, // назва прикладної області
    action: action, // назва кінцевого маршруту
    method: req.method, // тип HTTP методу
    path: req.url, // повний шлях запиту
    ip: req.ip, // ip запиту
    requestId: v4(), // унікальний ідентифікатор запиту
    schema: this._schemas, // знімок бізнес-схеми
    language: contextLanguage, // тип мови перекладу, яким потрібно надавати текстові відповіді
};

public:route​

Тип приватизації не передбачає, що запити є приватними та не перевіряє наявності токенів доступу, оскільки передбачається, що запити з типом public:route використовуються для публічних маршрутів, такі як головні сторінки, опис товарів тощо.

private:user​

Тип приватизації передбачає, що запиту є приватними, та потребують jwt токену доступу користувача - x-user-access-token, який надається в двох випадках:

  • при авторизації - входу в веб-застосунок.
  • при оновлені, завдяки x-user-refresh-token.

При отриманні запиту, адаптер ядра обчислень спробує знайти http заголовок x-user-access-token, якщо такий заголовок відсутній адаптер поверне наступну відповідь з http статус кодом 400:

{
  "type": "FAIL",
  data: {
    "code": "1002.1001",
    "message": "Authorization failed: miss required `x-user-access-token` http header"
  }
}

Якщо ж заголовок наявний, але термін його терміну дії сплинув, адаптер поверне наступну відповідь з http статус кодом 401:

{
  "type": "FAIL",
  data: {
    "code": "1002.1002",
    "message": "Authorization failed: user access token has expired."
  }
}

Якщо ж заголовок наявний, а токен дійсний, то з наявного у jwt токену адаптер бере sessionId, який створюється при авторизації користувачем на вебзастосунку та отримує інформацію про сесію користувача в Redis сховищі, після чого кладе цю інформацію в контекст виконання

const context = {
    // ..
    user: {
        userId: userId  // унікальний ідентифікатор користувача
        sessionId: sessionId // унікальний ідентифікатор сесії користувача
        ...params // інша, супутня інформація, яка формується при авторизації користувача та визначається бізнес-логікою
    }
}
примітка

Подібний тип авторизації використовується на маршрутах доступу до профілю користувача, його особистих даних, ролі, а також та ін.

private:system​

Тип приватизації передбачає, що запиту є приватними, та потребують jwt токену доступу системи - x-system-access-token, який надається в трьох випадках:

  • при видачі токена напряму, прописуючи його в договорі чи ін.
  • при авторизації в вебзастосунку через маршрут, який описується в бізнес-схемі.
  • при оновлені, завдяки x-system-refresh-token.

При отриманні запиту, адаптер ядра обчислень спробує знайти http заголовок x-system-access-token, якщо такий заголовок відсутній адаптер поверне наступну відповідь з http статус кодом 400:

{
  "type": "FAIL",
  data: {
    "code": "1003.1001",
    "message": "Authorization failed: miss required `x-system-access-token` http header"
  }
}

Якщо ж заголовок наявний, але термін його терміну дії сплинув, адаптер поверне наступну відповідь з http статус кодом 401:

{
  "type": "FAIL",
  data: {
    "code": "1003.1002",
    "message": "Authorization failed: system access token has expired."
  }
}

Якщо ж заголовок наявний, а токен дійсний, то з наявного у jwt токену адаптер бере sessionId, який створюється при авторизації системи в веб-застосунку та отримує інформацію про сесію системи в Redis сховищі, після чого кладе цю інформацію в контекст виконання

const context = {
    // ..
    system: {
        systemId: systemId  // унікальний ідентифікатор системи
        sessionId: sessionId // унікальний ідентифікатор сесії системи
        ...params // інша, супутня інформація, яка формується при авторизації системи та визначається бізнес-логікою
    }
}
примітка

Подібний тип приватизації використовується на маршрутах доступу інтеграцій, та при побудові між системної взаємодії чи ін.

Опис приватизації​

Ми пропонуємо просунуто описувати структури обробників запитів для повного контролю TypeScript над кодовою базою бізнес-схеми.

  • Router
  • Types declaration
import { setRouter } from "@x-fiber/calculator";

import type { NBusUsersAgg } from "~BusinessAdmin/Domains/Types";

export const BusUsersAggRouter = setRouter<NBusUsersAgg.Routes>({
    "v1/getOne": {
        GET: {
        scope: "private:user",
        params: ["userId"],
        handler: async (request, agents, context) => {
                context.user.firstName // Поле сесії;
                context.user.lastName // Поле сесії;
            },
        },
    },
});

export type BaseSessionInfo = {
    sessionId: string;
    userId: string;
};

export type Routes = {
    "v1/getOne": {
        GET: NAbstractHttpAdapter.Handler<
            void // тіло запиту,
            void // параметри запиту,
            void // заголовки запиту,
            void // query параметри запиту,
            void // тіло відповіді запиту,
            void // заголовки відповіді запиту,
            BaseSessionInfo & { firstName: string; lastName: string } // тип сесії користувача,
            void // тип сесії системи,
            "private:user" // тип приватизації  "public:route" | "private:user" | "private:organization";
            >;
    };
};
