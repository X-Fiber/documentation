Репозиторій MongoDB

інформація

Репозиторій MongoDB відіграє ключову роль у системі, де використовується база даних MongoDB. Він є інструментом для взаємодії з колекціями даних, забезпечуючи можливість виконання різноманітних запитів до них. Репозиторій дозволяє відокремити логіку доступу до даних від бізнес-логіки, що сприяє підвищенню читабельності та розширюваності коду. Використання репозиторію MongoDB дозволяє ефективно керувати та оптимізувати роботу з базою даних, що є важливим аспектом будь-якої сучасної програмної системи.

Архітектура​

Кожен репозиторій та модель MongoDB мають бути зареєстровані в відповідній прикладній області, яка, в свою чергу, повинна бути зареєстрована в відповідному сервісі. Під час завантаження бізнес-схеми в ядро обчислень та при отриманні запитів адаптер ядра отримує від сервісу актуальний знімок структури бізнес-схеми. Цей знімок додається в контекст виконання запиту. Під час виконання запиту, якщо він містить посилання на репозиторій тієї ж або іншої прикладної області, яка також належить до цього ж сервісу, здійснюється пошук необхідної функції. Після цього до неї додаються потрібні аргументи, та виконується обробка запиту в базі даних MongoDB.

примітка

У цьому контексті, зʼєднання та робота провайдера залишаються у сфері відповідальності ядра обчислень, тоді як бізнес-логіка покладається на бізнес-схему.

Cклад​

Репозиторій відображає структуру даних у форматі "ключ-значення", де кожен ключ відображає назву функції, а йому відповідає власне сама функція. Цей підхід дозволяє зручно організувати та керувати функціональними можливостями системи, забезпечуючи швидкий доступ до потрібного функціоналу через ім'я функції як ключ.

type Handler<ARGS = void, RESULT = void> = (args: ARGS) => Promise<RESULT>;
type Handlers = Record<string, Handler>
type Provider = {
    // ... list of methods
}
type Agents = {
    // ... fnAgent, schemaAgent, etc.
}

type Structure<S extends Record<string, any>, T extends Handlers = Handlers> = {
    [K in keyof T]: T[K] extends (data: infer D, ...args: infer A) => infer R
        ? (provider: Provider, agents: Agents, data: D) => R
        : T[K];
};

де:

  • Handler - тип метода запиту до колекції MongoDB.
    • ARGS - тип аргументів методу, які можуть бути передані при його виклику.
    • RESULT - тип результату, який буде повернутий після виконання методу.
  • Handlers - тип переліку методів запиту до колекції MongoDB.
  • Provider - провайдер функціональності, який надає перелік методів по створеню запитів до колекції MongoDB моделі цієї ж прикладної області.
  • Structure - загальний опис кінцевої структури обробників.
    • S - тип структури моделі колекції MongoDB.
    • K - назва методу.
    • T - структура методу.
    • D - Тип даних, які будуть передані в метод, який дорівнює ARGS типу Handler.
    • R - Тип даних результату повернення з методу, який дорівнює ARGS типу Handler.
    • provider - провайдер функціональності, який дорівнює Provider.
    • agents - агенти функіональності.
    • data - Тип даних, які будуть передані в метод, який дорівнює ARGS типу Handler.
warning

Така типізація дозволяє описувати простий тип функції, визначаючи лише ті аргументи, які потрібно передати, а також тип результату, який функція має повернути. Додавання аргументів provider та agents до функції залишається відповідальністю ядра обчислень, оскільки робота provider залежить від з'єднання з базою даних та оркестрації цих з'єднань, а agents представляють набір функціональності, яка надається ядром обчислень.

Реалізація​

import { setMongoRepo } from "@x-fiber/calculator";

type User = { name: string, login: string }
type Repository = {
    create: (user: User) => Promise<void>
    getById: (id: string) => Promise<User | null>
}
type Provider = {
    // ... list of methods
}
type Agents = {
  // ... fnAgent, schemaAgent, etc.
}


export const repository = setMongoRepo<User & { _id: string }, Repository>({
    create: async (provider: Provider, agents: Agents, data: User): Promise<void> => {
        // ... handler logic
    },
    getById: async (provider: Provider, agents: Agents, data: string): Promise<User | null> => {
        // ... handler logic
    }
});

де:

  • Типи:
    • User - проста структура моделі бази даних MongoDB.
    • Repository - перелік методів по роботі з колекцією бази даних MongoDB.
    • setMongoRepo - вбудована функція з ядра обчислень, для опису переліку методів по роботі з колекцією.
    • Provider - тип провайдера з переліком методів по роботі з колекцією MongoDB.
    • Agents - перелік агентів функціональності
  • Структури:
    • repository - обʼєкт з переліком методів по роботі з колекцією бази даних, який дорівнює типу Repository.
      • create - метод створення документа в колекції бази даних MongoDB.
      • getById - метод отримання документа по його унікальному ідентифікатору.

Реєстрація​

Щоб успішно використовувати методи доступу до конкретної колекції MongoDB, кожен з них повинен бути зареєстрований в точці входу прикладної області entry point. Наприклад, для модуля BusUsersAgg, необхідно використовувати функцію setPointer для встановлення посилання на репозиторій MongoDB та інші компоненти модуля.

import { setPointer } from "@x-fiber/calculator";
import { BusUsersAggMongoRepo } from "./BusUsers.agg.mongo-repository";

type BusDomains = 'BusUsersAgg' | 'BusUsersAuthSpec'

export const BusUsersAggPointer = setPointer<BusDomains>("BusUsersAgg", {
  mongoRepo: BusUsersAggMongoRepo,
  // ... other documents
});
